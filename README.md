# app_snap

A homelab toolkit that produces clean, consistent local backups of Dockerized services — configs, appdata, and databases — into a single directory, so an offsite tool like Duplicati can ship that directory to cloud storage without ever wrestling with live databases.

## The problem

Pointing Duplicati (or any file-copying tool) straight at Docker appdata produces corrupt database backups. These tools read files block-by-block over time; a SQLite or Postgres file that's being written mid-copy represents no single point in time. On restore, that file is corrupt or requires recovery.

The fix is a two-layer approach:

1. **Dump layer (this project):** orchestrates per-service backups locally — dumps databases with their own native tools (`pg_dump`, `mariadb-dump`, etc.), optionally stops containers for a clean appdata copy, and rsyncs everything into a local backups directory.
2. **Offsite layer (Duplicati / rclone / etc.):** points at that backups directory and ships it to Google Drive, S3, Backblaze, etc. Versioning lives offsite; locally you keep one clean "latest" copy per service.

---

## How it works

```
per-service:
backup-<app>.sh  ──cron──▶  DB dumps + appdata mirror ┐
                             + restore-manifest.env    │
                                                       ▼
                             <BACKUPS_ROOT>/<app>/
                                                       ▲
restore.sh (interactive) ────reads manifests──────────┘
   ↳ safety snapshot → restore → verify → keep OR rollback

                              ▼
                    Duplicati → cloud storage
```

Each service gets its own self-contained **backup script**, generated by the wizard. That script:

1. Dumps any databases via `docker exec` (no DB clients needed on the host)
2. Optionally stops the app container for a clean appdata copy
3. Rsyncs appdata into the backups root, excluding cache/log noise
4. Restarts the container
5. Writes a `restore-manifest.env` recording everything `restore.sh` needs

A single **`restore.sh`** lives in the backups root and works across all services. It presents an interactive menu, takes a safety snapshot of the current live state, restores the chosen backup, asks if it worked, and either keeps the restore or rolls back automatically.

### Supported databases

| Type | Method |
|---|---|
| SQLite | Cold file copy (when `STOP=1`) or `sqlite3 .backup` (hot) |
| PostgreSQL | `pg_dump --clean --if-exists` via `docker exec` |
| MySQL / MariaDB | `mysqldump` / `mariadb-dump` via `docker exec`, auto-detected |
| MongoDB | `mongodump` via `docker exec` |

## Requirements

- Docker (Compose or standalone containers)
- Python 3 (for the generator wizard)
- `rsync` on the host
- `sqlite3` on the host only if using hot SQLite backups (`STOP=0`)
- No database clients needed on the host — dumps run inside the containers

## Quick start

### 1. Generate a backup script for a service

```bash
python3 backup-script-generator.py
```

The wizard asks about a dozen questions (app name, appdata path, backups destination, container setup, databases, excludes, cron schedule) and writes `backup-<app>.sh` — executable and ready to run.

### 2. Run a manual backup to verify

```bash
./backup-<app>.sh
```

Check the log at `<BACKUPS_ROOT>/<app>/backup.log`.

### 3. Schedule it

The wizard prints a ready-to-paste cron line at the end, e.g.:

```
0 3 * * *  /path/to/backup-myapp.sh >> /path/to/backups/myapp/backup.log 2>&1
```

Add it with `crontab -e`.

### 4. Point Duplicati at the backups root

Configure Duplicati (or your offsite tool of choice) to back up `<BACKUPS_ROOT>/`. Since all databases are already dumped to portable files there, Duplicati just copies static files — no torn reads.

## Per-service setup: the wizard

Run `python3 backup-script-generator.py` and answer the prompts in order:

| Prompt | Notes |
|---|---|
| App name | Used for directory names and log files |
| Appdata path | Absolute path to the live appdata on the host |
| Backups root | Where the mirror lives (e.g. `/mnt/array/backups`) |
| Container management | Compose project or standalone container(s) |
| Compose dir / service | Or comma-separated container names for standalone |
| Stop during copy? | Recommended (`Y`). Required for consistent SQLite. |
| Databases | Add as many as needed; SQLite, Postgres, MySQL/MariaDB, or Mongo |
| Exclude common junk | Skips `cache/`, `logs/`, `tmp/`, `*.log`, `*.tmp` |
| Additional excludes | Comma-separated rsync patterns |
| Cron schedule | Default: `0 3 * * *` (3 AM daily) |
| Output script path | Default: `./backup-<app>.sh` |

The generated script is fully self-contained — all config is baked in. To change settings, re-run the wizard or edit the variables at the top of the script.

## Restore guide

`restore.sh` lives in the backups root and handles all apps it finds there.

### Basic restore

```bash
bash /path/to/backups/restore.sh
```

Follow the interactive prompts:
1. Choose the app to restore
2. Choose scope: **all** (appdata + databases), **appdata only**, or **databases only**
3. Confirm — the script takes a safety snapshot of the current live state first
4. After restore, you're asked "does it work?" — answer `y` to keep it or `n` to roll back automatically

### Dry run

```bash
DRY_RUN=1 bash /path/to/backups/restore.sh
```

Prints every action without executing anything.

### What the safety snapshot does

Before touching anything, `restore.sh` dumps the current live databases and snapshots the current appdata to `.safety-<app>-<timestamp>/` inside the backups root. If you answer `n` at the verify prompt, it restores from that snapshot and removes it. If you answer `y`, it deletes the snapshot.

### Notes

- Postgres restores require the target database to already exist. If you're restoring to a fresh host, use the **ALL databases** option when setting up (uses `pg_dumpall`, which includes `CREATE DATABASE`).
- Excluded directories (caches, logs) are removed from appdata on a full restore — they're regenerable and weren't in the backup.

## Testing

The repo includes a crash-test harness (`crashtest.sh`) that spins up a 3-container stack (Alpine app + Postgres + MariaDB), runs a full backup, and exercises disaster recovery, rollback, and partial-restore scenarios. Run it on your Docker host:

```bash
./crashtest.sh
```

It's idempotent — resets the stack on each run.

## Design decisions

**Dumps via `docker exec`** — No DB clients needed on the host. The backup script dumps into the running container, so the container must be up at dump time. The script handles this by dumping before stopping the app container, and `restore.sh` explicitly starts DB containers before importing.

**Overwrite latest locally** — Each backup run overwrites the previous one. Versioning is the offsite tool's job. Keeps local footprint small.

**rsync `--delete-excluded`** — Keeps the backup mirror clean of files that excluded patterns should have blocked from earlier runs.

**Atomic `mkdir` locks** — macOS has no `flock`; `mkdir` is atomic everywhere. An `EXIT` trap cleans up the lock and restarts containers if they were stopped.

**Bash 3.2 compatible** — macOS ships bash 3.2 (no `mapfile`, no `${var,,}`, no associative arrays). Both `restore.sh` and generated scripts stay within 3.2.
